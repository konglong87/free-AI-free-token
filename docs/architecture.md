# Architecture

## 目标

本仓库服务三个目标：

1. 给用户查阅免费 LLM 供应商的官网、免费模型、接入地址和模型能力。
2. 给开发者提供清晰的接入地址和接入步骤。
3. 给 agent 用户提供 Claude Code、OpenClaw、OpenCode、Codex、Hermes 的固定接入说明。

## 核心设计

### 一个供应商一个目录

`providers/<provider-slug>/` 是供应商的唯一边界。所有和该供应商有关的信息都必须放在目录内，不在根 README 中堆大段细节。

这样做的好处：

- 信息隔离，避免供应商之间互相污染。
- 后续可以自动生成网站、表格、搜索索引。
- 每个供应商都能独立 review、更新、删除。

### Markdown 作为唯一资料载体

本仓库优先做免费供应商资料收集。所有信息都写在 Markdown 中，方便直接阅读和 review。

### 只保留公开可接入信息

每个供应商只收集这些内容：

- 官网和官方文档入口
- API key 获取入口
- 接入地址
- 免费模型
- OpenAI-compatible 状态
- 固定 agent / coding client 接入方式
- 核验日期和核验状态

不收录营销介绍、使用感受、作者信息、账号信息、截图中的敏感字段或任何非公开内容。

### 接入文档固定化

每个供应商都必须提供五类 agent / coding client 的接入文档：

- Claude Code
- OpenClaw
- OpenCode
- Codex
- Hermes

如果某个供应商暂时无法确认某客户端的接入方式，也必须保留对应文件，并写明 `当前状态: 待核验` 或 `当前状态: 不支持`。

## Provider Slug 规则

供应商目录名使用小写 kebab-case：

- 推荐：`google-ai-studio`
- 推荐：`openrouter`
- 推荐：`groq`
- 不推荐：`GoogleAI`
- 不推荐：`open_router`

## 文档状态

每个供应商文档都应该明确状态：

- `待核验`: 信息来自公开资料，但还没有亲测。
- `已核验`: 已经用 API key 或官方控制台亲测成功。
- `部分可用`: API 可用，但某些客户端接入未验证。
- `失效`: 免费模型或 API 接入已经失效，保留历史记录。

## 更新原则

1. 官方信息优先于博客、视频、社区帖子。
2. 当前控制台显示优先于旧文档。
3. 实测结果优先于口头描述。
4. 不确定就标记为 `待核验`。
